# AI Research Assistant — Multi-Agent RAG System

A multi-agent AI research assistant built for the Delphi GenAI Intern case study, using
real agent separation, decision-gated RAG, structured tool calling, and full
retrieval transparency.

## Why this design — key decisions

**1. Classification-gated RAG, not always-on RAG.**
The Coordinator classifies every query into `KNOWLEDGE_LOOKUP`, `COMPUTATION`,
or `GENERAL_REASONING` *before* any agent runs. Only `KNOWLEDGE_LOOKUP`
queries reach the Retriever Agent. This directly avoids the named pitfall
of "always calling RAG (no decision logic)." The classifier prompt is
explicitly given scope-awareness of the knowledge base's contents and is
conservative by design — verified empirically below, not just claimed.

**2. Retrieved context is a first-class field, not a side effect.**
Every response is shaped by `core.schemas.FinalResponse`, which carries
`retrieved_chunks: List[Chunk]` with chunk ID, source document, and
similarity score, all the way from the vector store to the CLI/UI. We
never just trust that "the LLM saw the context" — we *show* it.

**3. Real multi-agent separation, not prompt-chaining in disguise.**
- **Coordinator**: classification + routing only. No domain logic.
- **Retriever Agent**: RAG only. Refuses to answer from general knowledge
  if no chunk clears the similarity threshold.
- **General Agent**: reasoning + tool decision-making. Decides for itself
  (via a separate structured LLM call) whether Calculator or Web Search
  is needed, and with what input.

If you deleted either domain agent, the Coordinator could not answer
anything on its own — proof the separation is functional, not cosmetic.

**4. DuckDB as the vector store — an honest tradeoff.**
DuckDB has no native ANN index in the version we use. At this project's
scale (a handful of source documents, low hundreds of chunks),
brute-force cosine similarity over arrays stored in DuckDB is simple,
correct, and fast enough. We chose not to bolt on a dedicated vector DB
(Chroma/FAISS/Pinecone) because the case study explicitly asks for
DuckDB and our corpus size doesn't justify the added complexity.

**5. Local embeddings, not LLM-based embeddings.**
OpenRouter's free-tier models are chat-completion models, not embedding
endpoints. We use a local `sentence-transformers` model
(`all-MiniLM-L6-v2`) — free, fast, deterministic, and removes retrieval's
correctness from depending on an external API's uptime.

**6. AST-based calculator, not `eval()`.**
The calculator tool parses expressions with Python's `ast` module and
walks a whitelist of allowed operators, rather than calling `eval()` on
arbitrary text generated by an LLM. This closes an arbitrary-code-
execution risk that a naive implementation would introduce.

**7. Mocked web search is labeled, not disguised.**
The web search tool returns `"mocked": True` explicitly in its structured
output. The case study permits mocking this tool; we chose not to
pretend it's live, since a transparent mock is more honest than a fake
"fully working" claim.

**8. Model resolution via OpenRouter's free-models router.**
Free-tier model slugs on OpenRouter rotate (deprecations, rate limits)
faster than expected during development — we hit both a `404` (deprecated
slug) and `429` (upstream rate limit) on two different hardcoded model
names mid-build. Rather than chase a moving target, all three model
settings point at `openrouter/free`, OpenRouter's auto-selecting router,
which picks from whichever free models are currently live and support
structured outputs. This is a config-only change (`.env`), requiring zero
code changes — proof that externalizing model names as settings (rather
than hardcoding them) paid off the first time the underlying provider
landscape shifted under us.

## Tradeoffs

| Decision | Gain | Cost | Why it's right for this scope |
|---|---|---|---|
| DuckDB, brute-force similarity | Simplicity, matches spec | No ANN scaling | Corpus is small |
| LLM-based classification | Generalizes across phrasing | Extra latency/cost per query | Routing accuracy matters more than the few hundred ms |
| Star topology (all routing through Coordinator) | Easy to trace/test/debug | Less "autonomous" multi-agent feel | Predictability > cleverness for a graded system |
| Mocked web search | Demonstrates tool-calling cleanly | Not real-time data | Explicitly permitted by the case study |
| Two domain agents (not three) | Avoids cosmetic over-engineering | Slightly less "many agents" feel | GeneralAgent already has internal tool-decision logic; COMPUTATION and GENERAL_REASONING genuinely share that machinery |
| `openrouter/free` router vs. a pinned model slug | Resilient to upstream deprecation/rate limits | Slightly less predictable model identity per call | Free-tier availability rotates faster than a take-home's dev cycle; observed directly during development |

## Project structure

genai-research-assistant/

├── config/         # Centralized settings (pydantic-settings)

├── core/           # Classification, routing/coordination, shared schemas

├── agents/         # RetrieverAgent, GeneralAgent, BaseAgent

├── rag/            # Loader, chunker, embedder, DuckDB vector store, retriever

├── tools/          # Calculator (AST-safe), mocked web search

├── llm/            # OpenRouter client (structured + plain completions)

├── utils/          # Logging/tracing, custom exception hierarchy

├── scripts/        # Ingestion pipeline (separate from query-time serving)

├── evaluation/     # Eval dataset + runnable eval harness

└── ui/             # Optional Streamlit interface


## Setup

```bash
git clone <repo-url>
cd genai-research-assistant
python -m venv venv
venv\Scripts\activate          # Windows
# source venv/bin/activate     # Mac/Linux

pip install -r requirements.txt
copy .env.example .env         # Windows: copy   Mac/Linux: cp
# Add your OPENROUTER_API_KEY to .env

# Add documents to data/knowledge_base/ (PDF, .md, or .txt)
python scripts/ingest.py        # Build the vector index
python main.py                  # Run the CLI
# OR
streamlit run ui/streamlit_app.py   # Run the UI
```

## Logging / Tracing

All agent calls, classification decisions, retrieval steps, and tool
invocations are logged through `utils/logger.py`'s `trace_step` context
manager, with structured JSON output to both console and `logs/app.log`,
including per-step latency. This gives a full audit trail of any query's
journey through the system — and was directly useful during development
for diagnosing OpenRouter rate-limit and malformed-JSON issues without
guesswork.

## Known limitations

- DuckDB brute-force search will not scale past tens of thousands of chunks.
- Free-tier OpenRouter models occasionally return malformed JSON for
  structured calls (observed during development); `chat_structured()`
  retries with re-validation, but this isn't bulletproof against a model
  that's persistently non-compliant.
- We route through `openrouter/free` rather than a pinned model slug,
  since free-tier model availability rotates (we hit both a deprecated
  slug and an upstream rate limit on two different pinned models during
  development). This trades a small amount of model-identity predictability
  for resilience against provider-side changes.
- Web search is mocked, as explicitly permitted by the case study.
